# Cross-Backend Consistency Review: ROS2 / RTI / FastDDS

## Context

The three backends (ROS2, RTI, FastDDS) were developed at different times and the FastDDS backend was modeled after RTI. This review identifies inconsistencies in patterns, architecture, and code duplication that can be harmonized.

## Finding 1: RTI and FastDDS main.cpp are ~95% identical (MAJOR)

**Files**: `rti/src/main.cpp`, `fastdds/src/main.cpp`

The only differences are:
- Line 44: App description string
- RTI has `--qos-profile` CLI option (FastDDS doesn't)
- Lines 74-76: Component creation (different types)
- A couple of startup log strings

The **entire event loop** (~40 lines), **stats printing** (~20 lines), **shutdown sequence** (~10 lines), and **signal handling** are copy-pasted verbatim.

### Proposed fix: Extract shared standalone event loop into `app/`

Create `app/include/pj_bridge/standalone_event_loop.hpp` and `app/src/standalone_event_loop.cpp`:

```cpp
namespace pj_bridge {

struct StandaloneConfig {
  int port;
  double publish_rate;
  double session_timeout;
  bool stats_enabled;
};

// Runs the event loop until SIGINT/SIGTERM. Handles:
// - process_requests() every iteration
// - publish_aggregated_messages() at publish_rate
// - check_session_timeouts() every 1s
// - optional stats printing every 5s
// - ordered shutdown: clear callback â†’ unsubscribe_all â†’ middleware shutdown
void run_standalone_event_loop(
    BridgeServer& server,
    std::shared_ptr<SubscriptionManagerInterface> sub_manager,
    std::shared_ptr<WebSocketMiddleware> middleware,
    const StandaloneConfig& config);
}
```

Each backend's main.cpp becomes ~30 lines: CLI parsing + component creation + one function call.

**Files to create:**
- `app/include/pj_bridge/standalone_event_loop.hpp`
- `app/src/standalone_event_loop.cpp`

**Files to modify:**
- `rti/src/main.cpp` â€” simplify to CLI + component creation + `run_standalone_event_loop()`
- `fastdds/src/main.cpp` â€” same simplification
- `CMakeLists.txt` â€” add `standalone_event_loop.cpp` to `pj_bridge_app` sources

## Finding 2: Inconsistent `get_participant()` error handling (MINOR)

- **RTI** `DdsTopicDiscovery::get_participant()`: **throws** `std::runtime_error` if domain not found
- **FastDDS** `FastDdsTopicSource::get_participant()`: returns **nullptr** if not found

The FastDDS approach is better â€” the caller checks explicitly. RTI's `subscribe()` wraps everything in try-catch so the throw is caught, but returning nullptr/optional is more consistent with the rest of the API (`get_type()` and `get_domain_id()` both return optionals).

### Proposed fix

Change RTI's `get_participant()` to return `std::optional<dds::domain::DomainParticipant>` instead of throwing. Update `DdsSubscriptionManager::subscribe()` to check the optional (same pattern as the `get_domain_id()` check above it).

**Files to modify:**
- `rti/include/pj_bridge_rti/dds_topic_discovery.hpp` â€” change return type
- `rti/src/dds_topic_discovery.cpp` â€” return nullopt instead of throw
- `rti/src/dds_subscription_manager.cpp` â€” add nullptr check in `subscribe()`

## Finding 3: Inconsistent `unsubscribe()` to_close pattern (MINOR)

- **RTI**: Uses `std::optional<SubscriptionInfo> to_close;` + `if (to_close) { ... }`
- **FastDDS**: Uses `SubscriptionInfo to_close{}; bool should_close = false;` + `if (should_close) { ... }`

The RTI approach using `optional` is cleaner and more idiomatic.

### Proposed fix

Change FastDDS's `unsubscribe()` to use `std::optional` like RTI.

Note: FastDDS's `SubscriptionInfo` has raw pointers as members, so default construction followed by `std::move` works fine. But `std::optional` with `emplace(std::move(...))` is cleaner and avoids a default-constructed zombie state.

**Files to modify:**
- `fastdds/src/fastdds_subscription_manager.cpp` â€” change `to_close` + `should_close` to `std::optional<SubscriptionInfo>`

## Finding 4: RTI has `ref_count()`, FastDDS doesn't (MINOR)

RTI's `DdsSubscriptionManager` has a public `ref_count(topic_name)` method. FastDDS doesn't.

This method doesn't appear to be used by the core or tests â€” it was likely added for debugging.

### Proposed fix

Remove `ref_count()` from RTI (dead code). Or if desired for debugging, add it to both. I lean toward removing it since it's unused.

**Files to modify:**
- `rti/include/pj_bridge_rti/dds_subscription_manager.hpp` â€” remove declaration
- `rti/src/dds_subscription_manager.cpp` â€” remove implementation

## Finding 5: ROS2 subscription manager missing explicit copy-delete and destructor (MINOR)

- RTI and FastDDS both `= delete` copy/move constructors
- ROS2's `Ros2SubscriptionManager` and `GenericSubscriptionManager` don't (implicitly non-copyable due to mutex, but not explicit)

RTI and FastDDS subscription manager destructors explicitly call `unsubscribe_all()`. ROS2 doesn't â€” it relies on RAII (shared_ptr<GenericSubscription> destruction). This is fine but inconsistent.

### Proposed fix

Add `= delete` for copy/move to `Ros2SubscriptionManager` and `Ros2TopicSource` for consistency. Leave the destructor as-is since ROS2's RAII approach is correct.

**Files to modify:**
- `ros2/include/pj_bridge_ros2/ros2_subscription_manager.hpp` â€” add `= delete`
- `ros2/include/pj_bridge_ros2/ros2_topic_source.hpp` â€” add `= delete`

## Finding 6: ROS2 shutdown doesn't explicitly call middleware->shutdown() (MINOR)

- RTI and FastDDS both call `middleware->shutdown()` explicitly in the shutdown sequence
- ROS2 relies on BridgeServer destructor to handle WebSocket shutdown

### Proposed fix

Add `middleware->shutdown()` after `sub_manager->unsubscribe_all()` in ROS2's main.cpp shutdown sequence for consistency.

**Files to modify:**
- `ros2/src/main.cpp` â€” add `middleware->shutdown()` in shutdown sequence
