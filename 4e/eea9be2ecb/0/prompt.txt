# Fix 7 Bugs Found in Code Review

## Context

Code review revealed 7 bugs of varying severity. Two high-severity ref-counting issues (#1, #2) share a root cause: the session's subscription list can be out of sync with the subscription manager's ref counts. The plan fixes all 7, enhances the mock to detect ref-count bugs, and fixes two existing tests that test the wrong thing.

## Step 0: Enhance MockSubscriptionManager with ref counting

The current mock uses `std::set` ‚Äî it can't detect ref-count leaks or underflows. Replace with `std::unordered_map<std::string, int>` ref counting.

**File**: `tests/unit/test_bridge_server.cpp` (lines 206-247)

```cpp
// Replace std::set<std::string> subscribed_topics_ with:
std::unordered_map<std::string, int> ref_counts_;
bool underflow_detected_{false};
```

- `subscribe()`: `ref_counts_[topic]++`
- `unsubscribe()`: decrement; if already 0, set `underflow_detected_ = true` and return false
- `unsubscribe_all()`: clear `ref_counts_`
- Add helpers: `int ref_count(topic)`, `bool has_underflow()`, `bool is_subscribed(topic)` (returns `ref_counts_[topic] > 0`)

Also enhance `MockTopicSource` ‚Äî add topics to both `topics_` AND `MockSubscriptionManager::known_topics_` via a test helper so subscribe actually succeeds. Currently most subscribe tests fail because topics aren't in `known_topics_`.

## Step 1: Fix two incorrect existing tests

**File**: `tests/unit/test_bridge_server.cpp`

### Fix `UnsubscribeRemovesTopics` (line 855)

Currently subscribes to `/topic_a` and `/topic_b` which don't exist in the mock, so subscribe fails. The unsubscribe then has nothing to remove. The test is functionally identical to `UnsubscribeIgnoresUnsubscribedTopics`.

**Fix**: Add real topics to the mock before subscribing:
```cpp
mock_topic_source_->set_topics({{"/topic_a", "std_msgs/msg/String"}, {"/topic_b", "std_msgs/msg/String"}});
mock_sub_manager_->add_known_topic("/topic_a");
mock_sub_manager_->add_known_topic("/topic_b");
```
Then verify the subscribe succeeds, and the unsubscribe of `/topic_a` actually removes it (the `removed` array should contain `/topic_a`). Verify `/topic_b` remains subscribed.

### Fix `SubscribeIsAdditive` (line 897)

Same problem ‚Äî topics don't exist so subscribes fail. Cannot verify additivity.

**Fix**: Add topics to mock, verify that after subscribing to `/topic_a` then `/topic_b`, both appear in the session's subscriptions (verify via a get_topics or subscribe response that shows schemas for both).

## Step 2: Write failing tests for each bug

All tests go in `tests/unit/test_bridge_server.cpp` unless noted.

### Bug #1 ‚Äî Ref-count leak on disconnect during subscribe
Test `DisconnectDuringSubscribeCleansUpRefCounts`:
- Add topics `/t1`, `/t2`, `/t3` to mock
- Subscribe to all three
- Immediately simulate disconnect
- Verify `ref_count("/t1") == 0`, etc. (all zero)
- Verify `has_underflow() == false`

This test will **fail** with current code if disconnect races between subscribe and update_subscriptions. In the single-threaded test it may pass, but the structural fix (incremental session update) prevents the race.

### Bug #2 ‚Äî Ref-count underflow on partial resume
Test `ResumePartialFailureRemovesTopicsFromSession`:
- Add `/topic_a`, `/topic_b` to mock, subscribe, then pause
- Remove `/topic_b` from `MockTopicSource::topics_` (simulate topic disappearance)
- Resume ‚Äî `/topic_b` subscribe should fail
- Verify `/topic_b` is no longer in session subscriptions
- Disconnect ‚Äî verify `has_underflow() == false`

This test will **fail** with current code (underflow on cleanup).

### Bug #4 ‚Äî Unsubscribe ignores object format
Test `UnsubscribeAcceptsObjectFormat`:
- Add `/topic_a` to mock, subscribe successfully
- Unsubscribe with `[{"name": "/topic_a"}]`
- Verify `/topic_a` is in `removed` array
- Verify `ref_count("/topic_a") == 0`

This test will **fail** with current code (object entry silently skipped).

### Bug #6 ‚Äî Unbounded queue
Test goes in `tests/unit/test_websocket_middleware.cpp`:
Test `IncomingQueueBounded`:
- After initialize, push more than `kMaxIncomingQueueSize` messages via the IXWebSocket callback
- Verify only `kMaxIncomingQueueSize` are dequeued via `receive_request()`

This test will **fail** with current code (all messages enqueued).

### Bug #9 ‚Äî Division by zero
Test `InitializeRejectsZeroPublishRate`:
- Create BridgeServer with `publish_rate = 0.0`
- Call `initialize()` ‚Äî should return false

This test should already **pass** (the validation exists). But add it for documentation and verify entry points don't compute `1.0/rate` before calling initialize.

### Bug #15 ‚Äî Dead code
No test needed. Just remove the field.

### Bug #6 ‚Äî Bound the incoming queue
**Files**: `app/src/middleware/websocket_middleware.cpp`, `app/include/pj_bridge/middleware/websocket_middleware.hpp`
- Add `static constexpr size_t kMaxIncomingQueueSize = 1024;`
- In message callback: check `incoming_queue_.size() >= kMaxIncomingQueueSize`, if so log warning and return (drop message)

### Bug #7 ‚Äî Release clients_mutex_ before sending
**File**: `app/src/middleware/websocket_middleware.cpp`
- `send_reply()`: copy `shared_ptr<ix::WebSocket>` under lock, release lock, then `ws->send()`
- `send_binary()`: same pattern
- `publish_data()`: copy all `shared_ptr`s into a local vector under lock, release, then iterate and send

### Bug #1 ‚Äî Incremental session update during subscribe
**Files**: `app/include/pj_bridge/session_manager.hpp`, `app/src/session_manager.cpp`, `app/src/bridge_server.cpp`

Add to `SessionManager`:
```cpp
bool add_subscription(const std::string& client_id, const std::string& topic, double rate);
bool remove_subscription(const std::string& client_id, const std::string& topic);
```

In `handle_subscribe()`: after each successful `subscription_manager_->subscribe()`, immediately call `session_manager_->add_subscription()`. If `add_subscription` returns false (session gone), roll back with `subscription_manager_->unsubscribe()`.

Remove the batch `update_subscriptions()` call at end. Keep a separate loop for rate-only updates on pre-existing topics.

### Bug #2 ‚Äî Remove failed topics from session on resume
**File**: `app/src/bridge_server.cpp`, `handle_resume()` (lines 483-497)

Collect topics that fail to re-subscribe into `failed_topics`. After the loop, remove them from the session via `session_manager_->remove_subscription()` (or batch `update_subscriptions()`).
